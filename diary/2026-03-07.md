Explored how `bge-reranker-v2-m3` behaves in the actual pipeline. Built `notebooks/reranker_exploration.ipynb` to measure scores across controlled relevance tiers, then added a cross-lingual diagnostic to the EDA notebook.

## Key finding: the cross-encoder reranker is nearly blind to relevant German docs

The pipeline feeds English queries directly into the reranker against German documents. We measured the effect with a controlled experiment — same documents, three query language conditions:

| Pair | Exact-match doc score |
|---|---|
| EN-query / DE-doc (pipeline) | **0.022** |
| DE-query / DE-doc (oracle) | **0.787** |
| EN-query / EN-doc (control) | **0.841** |

The most relevant document scores 0.022 in the actual pipeline — indistinguishable from gibberish (≈0.000016). The language penalty is **+0.764** on the relevant doc and effectively zero on all irrelevant docs.

This is the worst possible failure mode: the model doesn't uniformly depress scores, it specifically suppresses the signal on semantically relevant pairs. The reranker has the knowledge (0.787 with a German query) but cannot bridge the token-level language gap at inference time.

## Why it happens

A cross-encoder sees query + document as raw concatenated tokens. Relevance is scored by attending across query tokens to document tokens. "detention" and "Untersuchungshaft" are completely different token IDs — no surface overlap, no trained cross-lingual attention bridge (fine-tuning was predominantly on English MS-MARCO pairs). The `[CLS]` relevance signal never fires.

Bi-encoders don't have this problem because each text is compressed into a vector independently — language is gone before comparison.

## Fix

Translate queries to German before passing to the reranker. The bi-encoder retrieval step handles English queries fine (cross-lingual embeddings). Only the reranker needs German input. This is based on one controlled toy example — need to validate on real val queries to know the actual recall impact.

## Other reranker observations

- Score distribution across all val queries (Example 3/4 in the notebook): gold citations don't always separate cleanly from hard negatives at a fixed threshold. Score gap between consecutive ranks is more reliable than an absolute cutoff.
- The 0.5 sigmoid threshold is not a reliable separator — score scale shifts per query.
