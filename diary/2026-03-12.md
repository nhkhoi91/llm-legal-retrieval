## How transformers work (conceptual)

A transformer encoder (like BERT/MiniLM) takes a sequence of tokens and produces a contextual vector for each token.

**Input:** `[num_tokens, d_model]` — each token is a learned embedding vector.

**Attention mechanism (per layer):**

```
Q = X @ W_Q  →  [num_tokens, d_k]
K = X @ W_K  →  [num_tokens, d_k]
V = X @ W_V  →  [num_tokens, d_v]

scores  = softmax(Q @ K^T / sqrt(d_k))  →  [num_tokens, num_tokens]
output  = scores @ V                     →  [num_tokens, d_v]
```

The `[num_tokens, num_tokens]` matrix is the attention weights — how much each token attends to every other token. Multiplying by `V` brings the output back to `[num_tokens, d_v]`. Shape is preserved through the transformer layers.

**Key point:** the word "bank" in "river bank" and "savings bank" gets *different* output vectors because the surrounding context changes the attention pattern. This is what "contextual embedding" means.

**Pooling (sentence embedding):**

```
[num_tokens, d_model]  →  mean over tokens  →  [d_model]
```

`SentenceTransformer.encode()` does tokenization + transformer forward pass + mean pooling + optional L2 normalization in one call. Output is one vector per document.

## UMAP

UMAP (Uniform Manifold Approximation and Projection) reduces high-dimensional vectors to lower dimensions while preserving neighborhood structure — docs that were close in high-dim space stay close after reduction.

- Not linear (unlike PCA) — captures curved manifold structure
- `n_neighbors`: controls how much local vs global structure is preserved
- `min_dist`: controls how tightly points are packed
- **For visualization:** `n_components=2`
- **For clustering (BERTopic):** `n_components=10` — higher dims preserve more structure for HDBSCAN

**Important:** UMAP visualization quality (how separated clusters look in 2D) does not reflect embedding quality for retrieval. Embeddings can be excellent for cosine similarity search while looking like a single blob in 2D UMAP — especially when the coloring axis (e.g., law domain) is not the dominant axis of variation in the embedding space.

---

## HDBSCAN

HDBSCAN finds clusters as **dense regions** in the embedding space, without requiring a fixed number of clusters upfront.

- `min_cluster_size`: minimum number of points for a region to be called a cluster. Smaller → more fine-grained topics + higher outlier %. Larger → fewer broad topics.
- Docs in sparse regions get assigned topic `-1` (outlier).
- Works well in conjunction with UMAP because high-dimensional spaces suffer from the curse of dimensionality — distances become less meaningful, making density estimation unreliable. UMAP → 10D first, then HDBSCAN.

**Tuning:** sweep `min_cluster_size` and pick based on outlier % + topic coherence.

## BERTopic pipeline

```
documents
    │
    ▼
[1] SentenceTransformer.encode()
    text → [n_docs, d_model]
    │
    ▼
[2] UMAP  (n_components=10, for clustering)
    [n_docs, d_model] → [n_docs, 10]
    │
    ▼
[3] HDBSCAN  (finds dense regions)
    [n_docs, 10] → topic ID per doc  (auto number of topics)
    │
    ▼
[4] CountVectorizer + c-TF-IDF
    ranks words distinctive to each cluster → topic labels
    "T3: tax | income | levy | deduction"
```

**CountVectorizer config:**
- `min_df=5`: drop words in fewer than 5 docs (noise/typos)
- `max_df=0.5`: drop words in >50% of docs (too common to be distinctive)
- `ngram_range=(1,2)`: include 2-word phrases like "criminal liability", more meaningful than single words
- `stop_words`: first-pass list of German function words + Swiss legal structural boilerplate (`artikel`, `absatz`, `satz`, `ziffer`, etc.) that appear frequently but carry no topic signal. This list is a starting point — need to do proper frequency analysis to find other high-frequency zero-signal words before finalising.

**c-TF-IDF** is like TF-IDF but treats each topic's documents as one big document, then scores words that are frequent within a topic but rare across all topics. This gives the most discriminative keywords per topic.

**Post-hoc topic reduction:** if HDBSCAN finds 80 raw topics, BERTopic can hierarchically merge similar ones down to a target count without re-running the expensive embedding/clustering steps.

## Why law domain coloring is a weak validation signal

Coloring the UMAP by law domain (1xx, 2xx, 3xx...) and expecting clear separation is misguided because:

1. Legal documents share enormous vocabulary across domains — "article", "obligation", "enforcement" appear in all domains
2. Law domain is a structural/administrative label, not a semantic one
3. The embedding model optimizes for semantic similarity, not domain classification

Better validation for retrieval:
- **Topic coherence**: do top keywords per topic make semantic sense?
- **Gold agreement (sec9)**: does a query get assigned to the same topic as its gold citation? This directly measures whether topics are useful for retrieval routing.
- **Retrieval metrics (MAP, NDCG)**: the ground truth — judge embeddings by retrieval performance, not 2D visualization.
